# Macro Plan for MoE 2026-2027

# Context

The current leading MoE models and training pipelines are designed with a focus on cloud deployment with NVIDIA high-end GPUs as the target hardware. Our in-house developed Cor3 chip enjoys superior power and compute efficiency but with assumptions on the model size, bit width and sparsity. This presents both a challenge and opportunity for us when designing MoE models for Cor3 chips. The challenge differs between the deployment on cloud P1 and on Tehama edge SoC.
For P1, the difficulty is to keep up with SOTA MoE architecture,  scale up all the compression technologies for various model sizes and compete with the (evolving) NVIDIA GPUs on throughput and total cost. For Tehama SoC, the challenge is that many of these design decisions for cloud application are not well suited to edge deployment and the opportunity lies in tailoring these models to enable efficient edge deployment.

# Working assumptions

Current assumptions used throughout planning which will be corrected and updated as new information becomes available.

* **A1:** Micro will provide overall 10x weight compression vs original 16 bit precision with less than 1% accuracy degradation.
* **A2:** 9-10MB SRAM available per Cor3 chip, Tehama will have 9MB and P1 will have 9-10MB for each chip which compose to give total available SRAM
* **A3: **Tehama will have 4GB of DRAM with up to 3GB available for a model parameters. P1 will have 8GB of DRAM for each chip so in typical 2xP1 deployment would have total 16GB available for model parameters
* **A4: **Conditional loading of weights is supported on Cor3
* **A5: **Tehama will primarily target batch_size=1 (or possibly 2/3) applications while P1 will use largest batches supported by memory and efficiency constraints to enable batching of user queries as used in similar cloud deployments
* **A6:** The abstract model of Cor3 performance that is to be delivered will accurately reflect the expected performance on Cor3. The compiler and hardware simulation are constantly change so while might not be practical to use for reliable numbers will be useful to answer specific questions around tensorization formats
* **A7: **Assume that tracking eval metrics Hellaswag, MMLU, ARC-C, GSM8K and Winogrande (used for current [baselines](https://wiki.labcollab.net/confluence/display/EDGEAI/Baselines+v2)) are adequate to assess impact on task performance

# Tenets

**T1 — Hypothesize and Measure, don't assume**
Settle architecture questions by measurement (PNNF → SPC), not prior expectation.

**T2 — Integrate what we prove; invent what we lack**
The deliverable is a composed recipe where methods are composed in a principled manner. Once a method (our invention or from the literature) has been proven it should be integrated. Left un-integrated it is stranded value.

**T3 — A measured win displaces; a projected one doesn't.** The incumbent ships and stays until a challenger *demonstrates* a better trade-off.

**T4 — Bracket, then converge.** Hold two baselines — one accuracy-feasible, one efficiency-feasible — and close the gap between them. The gap is our progress metric; a stuck gap means the target is currently infeasible and requires further invention or change to target.

# Goals

We define different goals for P1 and Tehama platforms, due to different requirement for cloud and edge applications. For Tehama the main goal is enablement rather than targeting specific improvement of metrics that aren't measurable (when the model doesn't fit). For P1, we compare against baseline models in terms of improvements in throughput and latency which translate to savings cost vs other cloud based offerings.

**Tehama goals**

1. Enable Qwen3-30B-A3B to run on Tehama which requires overall parameter reduction of 2x (**A1** and  **A3** )
2. TTFT reduction of 20-30% vs baseline model (as measured by abstract model)
3. Throughput increase of 20-30% vs baseline model (as measured by abstract model)

**P1 goals**

1. Enable GPT-OSS-120B equivalent model to run on P1 with 2 Cor3 chips which requires overall parameters reduction of 1.5x (**A1** and  **A3** )
2. Enable Qwen3-235B-A22B equivalent model to run on P1 with 4 Cor3 chips which requires overall parameters reduction of 1.5x (**A1** and  **A3** )
3. Target increase in throughput over baseline model of 30-50% as measured by abstract model ( **A6** ) (translates to reduction in dollars/token)
4. Target reduction in TTFT of 30-50% over baseline as measured by abstract model ( **A6** )

**Technology goals**
Develop suite of macro tools that combined deliver:

* Accuracy metrics of interest to be within 1% percentage point of baseline on Hellaswag, MMLU, ARC-C, GSM8K and Winogrande ( **A7** )
* Based on working assumptions **A1** and  **A3** , this will require parameter reduction of 2x for Tehama goal and 1.5x for P1 goal

# Risks and Mitigations

The goals set out in this project are very ambitious and carry many risks from technology uncertainty and dependency on other teams.

**Risks**

* **R1: **The large drops in accuracy that are observed in current macro baselines. The current status for Qwen3-30B-A3B is average drop of **4pp** at **1.5x** Macro compression. For Qwen3-235B-A22B average drop is less than 1% at 10% parameter reduction
* **R2: **Dependency on micro team to resolve significant accuracy degradation and deliver efficient training pipeline. For Micro,  **6-8pp** drop was observed for Qwen3-8B ([baselines](https://wiki.labcollab.net/confluence/display/EDGEAI/Baselines+v2))
* **R3:** Dependency on platform team and platform scaling issues. Complexity of adding and composing new features. Regressions and bugs introduced, particularly during migrations. Lack of bandwidth on platform team to assist with issues and add new features
* **R4: **Uncertainty about how accurately the abstract model models the true real world performance of Cor3. Maturity and capability of PNNF/SPC tooling to answer questions around tensorization formats

**Mitigations**

* **M1: **Current baselines do not include the full range of macro tools (embedding and layer pruning) and use relatively short healing runs, so expect some improvement. If accuracy at target compression is still not adequate will need to relax compression target by using more of the available DRAM, targeting smaller model size or relaxing accuracy constraints (either by allowing a larger margin or narrowing focus to specific tasks)
* **M2: **If reasonable accuracy cannot be maintained through micro pipeline, can explore using more higher precision data-types which will likely require reducing target model size or relaxing accuracy constraints (either by allowing a larger margin or narrowing focus to specific tasks)
* **M3: **Adoption ofPEFT methods to accelerate healing. More initial exploration and PoC work outside of Percipio2 to enable faster iteration and relieve pressure on platform team
* **M4: **Forcapability gaps we can investigate other means of approximating expected performance that does not use the compiler (potentially using cor3-inference-model or similar). For runtime concerns can look to do less dense design space scans and create surrogate model to get estimates for full space. We expect decode to be entirely memory bandwidth bound so reducing memory bandwidth requirement should directly translate to proportional speed gains

# Phase 1 focus on MLP blocks (Q3-Q4 2026)

In phase 1, we will establish an abstract model that characterize the performance of a given model architecture on Cor3. We will also setup the tooling to go from PNNF to SPC to enable measuring the performance of different architectures and configs for Cor3. We will use these to confirm the assertion that decode is almost entirely memory bandwidth bound and identify other key architectural parameters that impact the performance of MoE models on Cor3 (e.g. ratio of time taken for MLP blocks vs attention blocks during pre-fill and decode). In the meantime, we will investigate the composition of the existing Macro compression technologies aiming to provide recipes for bounding configurations for each of the target MoE models on the corresponding platform. We will also continue the current investigation for efficient routing strategies to further reduce the number of active parameters and other innovations to improve the performance of MoE models on Cor3. In additional new MoE specific cross layer parameter sharing schemes will be explored to provide better compression accuracy trade-off than existing methods. In particular, we are planning the following four work-streams.

## 1. MoE MLP block co-design

With input from hardware team will establish an abstract model for Cor3 that enables projecting the performance of different MoE architectures. The PNNF to SPC tools will also be used to explore particular questions and hypotheses around tensorizations and other architectural changes.  With these in place we will investigate:

* Different MoE MLP block architectures and their corresponding performance projections. Including:
  * Shared expert vs all routed
  * Expert dimension
  * Heterogeneous expert size
  * Tensorized experts
  * Discover hard limits on dot product size, input channels
* When using the PNNF to SPC tools randomly generated weights will be used in place of applying micro. These random weights will be sampled from the expected post-micro distributions (e.g. ~30% sparsity) to ensure they are representative

Deliverables:

* **D1.1:** Tooling to measure expected speed of given MLP design on target HW using PNNF to SPC tools
* **D1.2: **Abstract model for Cor3 performance and tooling to produce relative performance vs baseline
* **D1.3:** Architecture for "ideal" MLP block for MoE adopting qualitative (e.g. shared experts) and quantitative features (optimal expert size/ratio) from architecture exploration along with expected timings on target platforms. Here ”ideal“ is meant from the compute efficiency perspective

## 2. Compose existing established technologies to establish  tracking bounds

In this work-stream, we will work on a general compression strategy for MoE models by composing the existing compression methods. We will investigate the limit of each individual method and the most effective way to chain them together to achieve the desired architecture. One way to chain the tools together is to use the Calibration-Transform-Train framework, which ensures each pruning step is recovered to the original accuracy. Another is to extend the dynamic width and rank pruning to prune multiple axes simultaneously to meet a given target. Yet another approach is to use evolutionary search ([link](https://quip-amazon.com/OBzaAssBzs7Z/Evolutionary-search-for-optimal-compression)) and other architecture search methods. We need a principled way to find the correct technique, where to apply and to what amount in each step. While the focus will be on MLP layers, the proportion of compute the MLP layers take relative to attention layers (linear, quadratic and hybrid) will also be considered to assess the overall impact of each method.

Candidate tools to be composed include:

* MoEfication for dense model (current target baselines are MoE, but if this changes to dense this will be in scope)
* Moderate static embedding pruning (e.g. 2048 → 1792)
* Moderate layer pruning (e.g. remove 3-5 layers)
* Moderate expert pruning with REAP (e.g. remove 20% of experts)
* Moderate pruning of expert intermediate dimension with Nyström (cf=1.1/1.2) or dynamic pruning
* Apply BTT/TT to Nyström compressed experts
* Dynamic pruning for expert in/out and intermediate dimensions
* Look ahead routing that predicts active experts from the hidden state of previous layers

Deliverables:

* **D1.4:** Two composed configurations for each target model, one that’s accuracy feasible (accuracy within 1%) and one that’s efficiency feasible (parameter count that fits in available memory) that will serve as initial configurations for converging from following tenets **T3** and **T4**
* **D1.5:** Integration of composition method into Percipio with capability to apply to other models

## 3. Complete inflight work on variable expert per token

Standard MoE models use top-k routing with k experts invoked for each token and layer. Reducing the number of experts selected for each token can degrade model accuracy as some tokens require more experts. Allowing different numbers of experts to be used for each token can enable greater accuracy for the same overall compute budget. This is particularly beneficial to MoE models on Cor3 where more than the compute bottleneck there is a memory bandwidth bottleneck so reducing the average number experts per token can have significant benefit.

Work to date has demonstrated that switching to top-p routing with a static threshold can enable variable expert per token while maintaining the accuracy of standard top-k routing at the same compute budget. Work is ongoing to extend this to enable the same or greater accuracy at lower overall compute budget by exploring per token learned thresholds, directly predicting the number of tokens to route to and using a static threshold in normalized router logit space.

* **D1.6:** Integrated method to convert top-k model to model that uses a variable number of experts per token
* **D1.7:** Apply to pre-trained baseline model and baselines from work-stream 2
* **D1.8:** Report documenting impact on accuracy and timing using tooling from **D1.1 **and **D1.2**

## 4. Cross Layer Expert Sharing

In standard dense and sparse (MoE) models the number of parameters (total and active) scales with the depth of the model. There is no a priori reason this should be the case as parameters and width/depth serve different purposes. In the simplest interpretation parameters store information and depth adds processing and recursion (e.g. many algorithms apply the same operation many times, why not LLMs). Exploit redundancy in expert parameters across layers to significantly reduce parameter overhead and thus lead to more efficient inference.

* Enable the same experts to be used from different MLP layers via single universal expert pool or private local experts along with a universal pool that enables structured sharing
* Starting with pre-trained network, try to identify experts that have
* Explore the use of adaptors and latent projectors to enabling more sharing of experts between layers (Mixture of Cores)

Deliverables:

* **D1.9: **Integrated method that starts from standard pre-trained top-k model, identifies candidate experts or expert constituents, composes these and trains to enable effective sharing across layers. Possibly adaptors or latent projectors if shown to add benefit
* **D1.10: **Methods applied to baseline and reference bound models to provide accuracy numbers at different levels of sharing and corresponding timing results using **D1.1** and **D1.2**
